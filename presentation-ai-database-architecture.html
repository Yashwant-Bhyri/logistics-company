Database-Centric Architecture: AI Assistant & Ops Communication Center

Normalized MySQL is the source of truth. The LLM never generates arbitrary SQL — it routes to parameterized query families and summarizes retrieved relational facts.

Flow: Browser → REST → Flask core → Parameterized SQL / Transactions → MySQL   |   Response path reverses · LLM calls (OpenRouter) only after structured retrieval

Layer 4 — MySQL · Database LOGISTICS_COMPANY · relational + transactional

Normalized schema (3NF operational core). Ops messages may use SUPPORT_MESSAGE (ensured at runtime). All AI “intelligence” is grounded in rows returned by explicit SQL paths below.

A) READ — parameterized & joins
• shipment context: C_ORDER ⨝ CUSTOMER (parties)
  ⨝ ORDER_UPDATE ⨝ PAYMENT ⨝ DELIVERY ⨝ DRIVER …
• admin intents: assignments, routes, drivers,
  vehicles, condition_reports, overrides (LIMIT + filters)
• ops inbox/feed: SELECT … FROM SUPPORT_MESSAGE
  WHERE participant_* · ORDER BY message_id
• complaint queue: patterned filters on body/subject
B) WRITE — transactional rows
• INSERT SUPPORT_MESSAGE (thread, broadcast, logs)
• commit / rollback on multi-step mutations
• demo seed guarded by EXISTS checks
C) ANALYTIC-SQL (embedded OLTP analytics)
• horizon windows, OPEN vs RESOLVED complaint counts
• severity buckets, aging_histogram (time deltas)
• root_cause aggregates, driver_impact joins
→ SQL aggregations over normalized + message store
D) Integrity & design tie-in
• PK / FK (e.g. C_ORDER.sender_id → CUSTOMER)
• indexes recommended on participant keys + message_id
• course topics: relational model · ER mapping ·
  SQL · transactions · indexing · analytic queries

Differentiator for DB course: The architecture separates stored relational state (MySQL), audited parameterized retrieval (Python-called SQL templates), and non-authoritative narration (LLM). This mirrors enterprise “semantic layer over warehouse” patterns at app scale — without conflating analytics SQL with generative SQL.
